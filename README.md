##バックグラウンドと課題

　約100名規模の部活動における既存のGoogle Calender運用の限界を突破するために開発。


##現行運用の課題点

　・反映遅延によるダブルブッキング：Googleカレンダーの同期ラグ（10分以上）による予約重複の頻発。

　・同時アクセス制限：100名規模の一斉アクセスによるGoogleアカウントの認証ブロック。

　・複雑な部内ルールの形骸化：ライブ日程に合わせた「予約解禁日」の管理が手動であり、ルール違反が絶えない。

##解決アプローチ

　・Realtime Sync：Supabase Realtime (WebSocket) を採用し、予約状況を全端末へ0.1秒以内に反映。

　・独自認証基盤：各部員に固有IDを付与。Googleアカウントの共有を廃止し、アクセス制限を回避。

　・動的バリデーションエンジン：ライブ日程から逆算した「予約権限」をシステム側で強制。


##予約バリデーション・ロジック

ライブ開催日を `T` とした場合の予約可能期間を厳格に管理する。

|      予約種別      | 予約開始日 | 予約終了日 |          備考         |
| **　バンド練習　** | `T - 30日` | `T (当日)` | ライブ出演バンド優先枠 |
| **　　個人練習　** | `T - 14日` | `T + 7日`  | パート別（Dr/Gt/Ba等）の枠管理 |


##技術スタック

　・Backend: Python (FastAPI)
        Pydanticによる厳格な型定義と、非同期処理による高スループットな予約処理。
　・Frontend: React (Next.js / Tailwind CSS)
        スマホ利用を前提としたレスポンシブデザイン。SWR を活用した状態管理。
    Database: Supabase (PostgreSQL)
        Row Level Security (RLS) による部員間のデータ保護と Realtime 通知。
    External API: Google Calendar API
        既存の外部カレンダーとの参照用同期（片方向）。

##ポイント

スタジオの機材をパートごとに分解することで、スタジオの稼働率を最大化した。
